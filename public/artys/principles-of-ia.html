Principles of information architecture

Peeling the "why" layers of the onion slowly away to begin revealing the "how" of design-bias product decisions.

Aug 12, 2020

Information architecture (IA) is a discipline that creates relationships between all core decisions to creating product experiences.

IA is also a critical community-based skill that is required to transition an organization to a more mature design state.

This is true even with design systems. IA enables a design system to transition from mere component libraries toward the efficient cultural collaboration of how teams create products together.

So, in an effort to articulate and raise awareness of a standard level of IA quality that embeds itself into products within organizations, I've written five principles of IA.

1. Minimize personas

Any product should strive to define and scope itself to a minimum number of core personas. Start at one, and only increase as absolutely necessary.

2. Minimize user goals

Any product should strive to define and scope itself to a minimum number of user goals per persona. Start with one, and only increase as absolutely necessary.

3. Minimize user goals per page

Any product should strive to define and scope itself to a minimum number of user goals per page. Start with one, and only increase as absolutely necessary.

4. Content follows user goals

Product content should reflect personas' goals in a way that answers “what” information helps a person achieve goal success.

5. Sequence content over time

Product content should be sequenced over time in a way that answers the “when” and “where” information best helps a person achieve goal success.

As I alluded to in the beginning, these principles support and use the design-bias decisions for creating products. In using these as common vocabulary terms, project teams enable themselves to ensure a fair distribution of power as they create products.

This benefit is extended, and thus amplified, when considering the promotion of these prinicples through a design system. Now, every project team that uses these principles are setting themselves up with high-level decision guiding points that focus and improve product quality.
